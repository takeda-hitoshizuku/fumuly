# AIネイティブタスク管理の思想

## これは設計ではなく、結果的にそうなった話

最初に断っておくと、「AI駆動開発に最適化されたタスク管理システムを設計しよう」と思って始めたわけではない。

始まりは、タスク管理ツールを自分で開発しようとしたこと。要件を考えて、画面を設計して、データベースを組んで——という普通の開発プロセスに入ろうとした。でもそこで気づいた。

**「これ、拡張機能で良いんじゃね（暴論）」**

タスク管理に必要な機能を分解してみたら、ファイルの読み書きとフロントマターの解析だけで大部分が成り立つ。であれば、Claude Codeの拡張として動かせばいい。SaaSを作る必要すらない。

フロントマターも、最初から「AIに読ませるため」に設計したわけではない。**完全に自分用**だった。タスクの状態を管理するために `status: in-progress` と書き、依存関係を忘れないために `depends_on: [task-002]` と書いた。それが結果的に、AIが機械的にパースできるフォーマットそのものだった。

つまりこの仕組みは：
- 人間が自分のために書いたフォーマットが
- たまたまAIにとっても最高に読みやすいフォーマットだった
- それに気づいて「管理させられるじゃん」となった

**狙って作ったのではなく、使ってみたら噛み合った。** Fumulyで実際に42タスクを運用してきた中で見えてきた、AI駆動開発におけるタスク管理の考え方をまとめる。

---

## 基本構造

1ファイル = 1タスク（`tasks/task-XXX.md`）。YAML frontmatter + Markdown本文という、誰でも知ってるフォーマットの組み合わせ。

### frontmatterが提供する機能

| フィールド | 機能 | 用途 |
|-----------|------|------|
| `id` | 一意識別子 | タスク間の参照に使用 |
| `title` | タイトル | 一覧表示・検索 |
| `parents` | 複数カテゴリ | `[UI/UX, 機能]` のように複数の関心事を表現 |
| `status` | 6段階ステータス | idea → waiting → in-progress → done/omit/blocked |
| `priority` | 4段階優先度 | critical / high / medium / low |
| `depends_on` | 依存関係 | `[task-002, task-003]` で前提タスクを定義 |
| `this_week` | 週次スコープ | 今週やるかどうかのフラグ |
| `completed_at` | 完了日 | 時系列での振り返り |
| `progress` | 進捗率 | 0-100の数値 |
| `note` | 一行メモ | frontmatterだけ見て概要を把握 |
| `estimated_hours` | 見積もり | 工数管理 |

### Markdown本文の役割

- **未完了タスク** → 要件定義書（背景・ユースケース・実装案・検討した選択肢）
- **進行中タスク** → 作業コンテキスト（今何をしているか）
- **完了タスク** → 設計書（作業メモ・実装内容・見送った案とその理由）

タスクの「完了」は情報の消滅ではなく、**設計ドキュメントへの昇格**。

### タスクのライフサイクル

```
idea → waiting → in-progress → done
 (要件の種)   (要件定義書)    (作業中)     (設計書)
                ↘           ↕
               omit       blocked
           (やらない理由書)  (外因による待機)
```

どのフェーズでも本文に情報が蓄積され、完了後もプロジェクトの知識ベースとして機能し続ける。omitされたタスクも「なぜやらないか」の記録として残り、状況が変われば復活できる。blockedは外因（Stripeの審査待ち、他サービスの承認待ち、ボスの確認待ち等）による一時停止で、解除されればin-progressに戻る。

## 既存ツールにない特徴

### 複数親カテゴリ

`parents: [セキュリティ, UI/UX]` で1タスクが複数の関心事に属する。多くのツールは1カテゴリか、タグ機能があっても検索が弱い。これがあるから「セキュリティ関連のタスク」でも「UI/UX関連のタスク」でも、どちらからでもそのタスクにたどり着ける。

### グルーピング能力

| 軸 | 方法 |
|----|------|
| カテゴリ別 | `parents` でgrep |
| ステータス別 | `status` でgrep |
| 優先度別 | `priority` でgrep |
| 依存関係別 | `depends_on` で芋づる式に辿る |
| 週次スコープ | `this_week` でgrep |
| 時系列 | `completed_at` でソート |

これらを自由に組み合わせてクロス集計できる。「セキュリティ関連で未完了のhighタスクは？」のような質問にAIが数秒で答えられる。

### AIネイティブ

人間もAIも同じ方法（ファイルの読み書き）でアクセスする。SaaSツールのようにGUIとAPIで別々の入口を用意する必要がない。

### CLAUDE.mdによる規約定義

フォーマット・カテゴリ一覧・運用ルールをCLAUDE.mdに書くだけで、AIが従う。設定画面もプラグインも不要。

### ゼロ依存

プレーンテキストなのでベンダーロックインがない。Git管理でバージョン履歴も自動的に残る。

## AIがタスクをどう読み、どう探すか（技術的な裏側）

AIコーディングエージェント（Claude Code等）は、内部的に以下の3種類のツールを使ってファイルシステムを操作する。SaaSのAPIではなく、開発者がターミナルで行う操作と本質的に同じこと。

### ファイル操作はAIの母語

人間が `grep "^status: waiting" tasks/` と打とうとすると、まずコマンドの構文を思い出し、正規表現を組み立て、出力をパイプで繋いで——という手順が必要になる。慣れた開発者でも数十秒かかる。

AIにとって、これは**母語**。Glob/Grep/Readはエージェント実行環境に最初から組み込まれたネイティブ操作で、考える時間ゼロで発行できる。しかも複数のクエリを並列で同時に走らせられる。人間がgrepのオプションを調べている間に、AIは既に全タスクをスキャンし終わっている。

SaaSのGUIは「コマンドが難しい」という人間の課題を解決するために作られた。この仕組みは同じ課題を別の方向から解決した——**コマンドが得意な奴に任せればいい。**

言い換えれば、フロントマターは人間の意図をAIが読める形式に変換する**翻訳機**。「自分が使いやすいタスク管理を作っていたつもりが、なぜかAIとの翻訳機を作っていた」というのが、この仕組みの正体かもしれない。

### AIが使う3つの基本ツール

| ツール | 対応するコマンド | 用途 |
|--------|-----------------|------|
| **Glob** | `find`, `ls` | ファイル名のパターン検索。`tasks/task-*.md` で全タスクファイルを一覧取得 |
| **Grep** | `grep`, `rg` | ファイル内容の正規表現検索。frontmatterの特定フィールドを横断検索 |
| **Read** | `cat` | ファイル内容の読み込み。タスク本文の詳細を取得 |

重要なのは、**この3つだけで全操作が完結する**こと。専用のクエリ言語もAPI認証も不要。

### 実際の動き：全体把握

AIがまず最初にやることは、Globでタスクファイルの一覧を取得すること。

```
Glob("tasks/task-*.md")
→ task-001.md, task-002.md, ... task-042.md （42件）
```

次にGrepでfrontmatterだけを一括スキャンする。

```
Grep("^status:", path="tasks/")
→ task-001.md: status: done
→ task-005.md: status: waiting
→ task-006.md: status: omit
→ task-014.md: status: waiting
→ ...（42件分のステータスが一瞬で返る）
```

42ファイルあっても、Markdown本文は読まない。frontmatterだけなら数秒で全タスクの状態を把握できる。**必要なものだけ深掘りする**のがポイント。

### 実際の動き：絞り込み

「セキュリティ系で未完了のタスクある？」と聞かれたら、AIは内部的にこう動く。

```
Grep("^parents:.*セキュリティ", path="tasks/")
→ task-004.md: parents: [セキュリティ]
→ task-007.md: parents: [セキュリティ]
→ task-015.md: parents: [セキュリティ]
→ task-018.md: parents: [セキュリティ, UI/UX]
→ task-034.md: parents: [セキュリティ]
→ task-042.md: parents: [セキュリティ, バグ修正]
```

ここで6件ヒット。次にこれらのステータスを確認して、`status: done`以外のものを抽出する。人間がダッシュボードでフィルタする操作を、AIはGrepで行う。結果は同じだが、APIキーもOAuth認証も不要。

**そしてここが重要：これらのツール呼び出しは並列実行できる。** Glob、Grep（セキュリティ検索）、Grep（ステータス検索）を同時に走らせて結果を突き合わせる。SaaSのAPIだとリクエストを順番に送る必要があるが、ファイルシステムは並列アクセスに制限がない。

### 実際の動き：深掘り

絞り込みの結果、気になるタスクが見つかったらReadで本文を読む。

```
Read("tasks/task-034.md")
→ frontmatter + Markdown本文がすべて返る
→ 「task-027と合わせてプライバシーリスクを軽減」という記述を発見
```

ここで本文中の `task-027` という文字列を見つけたら、次はそのファイルを読みに行く。

```
Read("tasks/task-027.md")
→ 会話履歴の保持期間ポリシーの設計書が出てくる
→ さらに task-021（プライバシーポリシー）への言及を発見
```

これが**芋づる式探索**。`depends_on` と本文中の `task-XXX` 参照をリンクとして辿る。人間にはただの文字列だが、AIにとっては**実質的なハイパーリンク**。

### ツール呼び出しの実行環境

これらのツールは、AIコーディングエージェントの実行環境によって提供される。

```
ユーザーの発話
  ↓
AIエージェント（Claude等）
  ↓ ツール呼び出し
┌─────────────────────────┐
│ エージェント実行環境      │
│  Glob → ファイルシステム  │
│  Grep → ファイルシステム  │
│  Read → ファイルシステム  │
│  Edit → ファイルシステム  │
│  Bash → シェル            │
│  Write → ファイルシステム │
└─────────────────────────┘
  ↓ 結果
AIが結果を解釈して回答
```

SaaSツールだと、このフローが以下に変わる：

```
ユーザーの発話
  ↓
AIエージェント
  ↓ MCP接続 or API呼び出し
┌─────────────────────────┐
│ 外部サービス              │
│  OAuth認証                │
│  API rate limit           │
│  JSONレスポンスのパース    │
│  ページネーション処理     │
└─────────────────────────┘
  ↓ 結果
AIが結果を解釈して回答
```

間に挟まるレイヤーが増えるほど、遅くなるし壊れやすくなる。ファイルベースだとこの中間レイヤーが**ゼロ**。

### 完了タスクを設計書として参照

新しいタスクに取り組む時、AIは関連する完了タスクをReadで読む。例えばtask-041（アーカイブ機能）に着手する際：

```
Read("tasks/task-027.md") → 会話履歴の件数ベース削除の実装パターンを参考に
Read("tasks/task-024.md") → select最適化の方針との整合性を確認
Read("tasks/task-021.md") → プライバシーポリシーとの法的整合性を確認
```

**完了タスクの作業メモが、次のタスクの設計判断に直接使われる。** SaaSツールの完了タスクはアーカイブされて二度と見られないが、この仕組みでは蓄積された知識として活用し続ける。

### セッションを跨いだ記憶の代替

AIは毎回セッションがリセットされる。でもタスクファイルをGrepとReadで読めば：

- **何が終わっていて**（`Grep("^status: done")` → 39件）
- **なぜその設計にしたか**（`Read` で本文の検討経緯を取得）
- **何がまだ残っていて**（`Grep("^status: waiting")` → 2件）
- **何に依存しているか**（`Grep("^depends_on:")` で依存関係マップ）

がすべてわかる。**タスクファイルがAIの長期記憶として機能している。**

### タスクの詳細がAIの探索コストを下げる

タスクに詳細が書いてあっても、AIは結局コードを見に行く。タスクの記述と実際のコードにズレがある可能性があるし、実装する時にはファイルの正確な構造（行番号、変数名、import文）が必要だから。

ただし、**どこを見に行けばいいか**がタスクでわかる。

```
タスクなし: Glob → Grep → Grep → Grep → Read → Read → Read（探索）
タスクあり: Read → Read（直行）
```

タスクに「`app/api/documents/route.ts` にバリデーションを追加した」と書いてあれば、プロジェクト全体を手当たり次第にGrepする必要がなく、いきなりピンポイントでReadできる。タスクの詳細は「読まなくて済む」ではなく**「迷わなくて済む」**効果。

### タスクが「正しさ」の比較対象になる

タスクがない場合、AIが「ちゃんとやったか」を検証する手段はgit diffとチャット履歴しかない。git diffは「何を変えたか」を記録するが、「何を変えるべきだったか」は記録しない。チャット履歴はセッションが切れたら消える。

```
タスクなし: コード変更 → 何と比較する？ → 比較対象がない
タスクあり: タスク（要件）→ コード変更 → タスクと突き合わせて検証
```

タスクの本文に「やるべきこと」が書いてある。完了後にそれと「やったこと」を突き合わせれば、抜け漏れに気づける。`progress: 100` にしていいかの判断材料がタスク自体にある。**整合性チェックという概念自体が、タスクがあって初めて成り立つ。**

## AIが関連タスクの更新漏れに気づける仕組み

例えばtask-041（アーカイブ機能と30日自動クリーンアップ）を実装した時：

1. task-041の本文に「30日でデータ削除」という仕様がある
2. `parents: [UI/UX, 法的文書]` → 法的文書カテゴリから関連タスクを探す
3. task-021（プライバシーポリシー修正）を見つける → 完了済みだが、**当時の修正内容にはまだ30日ルールが反映されていない**
4. task-027（会話履歴の保持期間）も見つける → こちらも関連

この時点で「プライバシーポリシーに30日自動削除の記述を追加する必要があります」と提案できる。

これができるのは、完了タスクの本文に**何を修正したか**が残っているから。task-021を読めば「第6条にアカウント削除の記載を追加した」ことがわかるので、「でも30日自動削除のことはまだ書かれていない」という差分に気づける。

## AIがタスクを追加する時の思考プロセス

「セキュリティの向上をタスクに切って」と言われた時の動き方：

### Step 1: 既存のセキュリティタスクを全部読む

`parents` に `セキュリティ` を含むタスクを探す。

- task-004（セキュリティ）→ 完了済み。何をやったか本文で確認
- task-007（セキュリティ）→ Google認証追加。完了済み
- task-015（セキュリティ）→ 完了済み。内容確認
- task-018（セキュリティ, UI/UX）→ パスワードリセット。完了済み
- task-034（セキュリティ）→ 暗号化。完了済み

**完了タスクの本文を読んで、「何が既に対策済みか」を把握する。** これをやらないと、既にやったことを重複して提案してしまう。

### Step 2: 関連カテゴリも横断チェック

セキュリティに関係しそうな他カテゴリも見る。

- task-021（法的文書）→ プライバシーポリシー修正。認証データの削除に言及
- task-027（法的文書, バックエンド）→ 会話履歴の自動削除。暗号化と併用
- task-041（UI/UX, 法的文書）→ 30日自動クリーンアップ

**`parents` の複数カテゴリ機能があるから、カテゴリを横断して関連タスクを発見できる。**

### Step 3: コードベースの現状を確認

タスクだけでなく、実際のコードも見る。API認証、RLS、入力バリデーション、環境変数の扱いなど。

### Step 4: 「まだやっていないこと」を特定

既存タスクの完了内容（設計書）とコードの現状を突き合わせて、**差分＝やるべきこと**を導き出す。

### Step 5: タスクファイルを作成

```yaml
---
id: task-043
title: APIレート制限の強化
parents: [セキュリティ, バックエンド]
status: waiting
priority: medium
depends_on: [task-034]
this_week: false
completed_at: null
progress: 0
note: "解析APIのレート制限を強化。既存のIP制限に加えユーザー単位の制限を追加"
estimated_hours: 2
---
```

ポイント：
- `parents: [セキュリティ, バックエンド]` → 複数カテゴリで適切に分類
- `depends_on: [task-034]` → 暗号化タスクとの関連を明示。次にこのタスクを拾ったAIが「task-034で暗号化済みだからそこは考慮不要」と判断できる
- 本文にはユースケース・背景・実装案・検討した選択肢 → 未完了タスク＝要件定義書になる

## 重複コードの防止

### なぜ重複が起きるか

バイブコーディングもコンテキストコーディングも、AIは**今見えているファイルだけ**で判断する。「この処理、別のファイルにもう書いてあるよ」という情報がないから、同じようなバリデーション、同じようなエラーハンドリング、同じようなAPIコールを何度も書いてしまう。

### タスクファイルがどう効くか

「書類保存時にバリデーションを追加して」と指示された時：

1. task-019（DB操作のエラーハンドリング）を読む → 「各所で `{ error }` を確認し、失敗時にユーザーへ通知」というパターンが既にある
2. task-023（JSON.parseのtry/catch）を読む → エラーハンドリングの日本語メッセージ方針がわかる
3. task-025（重複スキャン検知）を読む → 保存前チェックのパターンが既にある

**既存のパターンに合わせる判断ができる。** 新しい関数を書くのではなく、既存のやり方を踏襲する。

### ゼロにはならない理由

- タスクの本文に書かれていない暗黙的なパターン（コードの書き方の癖など）は拾えない
- タスク化されていない小さな実装（ユーティリティ関数など）は見落とす
- AIのコンテキストウィンドウの限界で、全タスクを一度に読めない場合がある

### 限りなく抑えるための条件

完了タスクの本文に以下が書かれていれば、重複はかなり抑えられる：

1. **どのファイルに何を実装したか**（実装箇所）
2. **どういうパターンで書いたか**（設計判断）
3. **なぜその方法を選んだか**（見送った案）

**完了タスクの品質 = 重複防止の精度。** 雑に「Done」とだけ書いてあるタスクだと効果は薄いが、作業メモがしっかり書かれていれば、重複はかなり抑えられる。

## コンポーネント思想・トークン思想との接続

タスク管理が「なぜ作ったか・何を作ったか」の記録なら、コンポーネント思想は「どう作るか」のルールブック。この2つは補完関係にある。

### コンポーネント思想

UIの部品の選定・使い方ルール。

- 「Buttonは`shadcn/ui`の`Button`を使う。独自ボタンは作らない」
- 「モーダルは`AlertDialog`で統一。`Dialog`は使わない」
- 「フォームは`react-hook-form` + `zod`で統一」

これがあると、AIが新機能を実装する際に既存コンポーネントを再利用する。ないと、同じようなボタンやダイアログを毎回ゼロから書いてしまう。

### プロップス

コンポーネントの振る舞い・バリエーション。

- `variant="destructive"` → 削除系のアクション
- `size="sm"` → コンパクトな表示

shadcn/uiが定義済みなので、プロジェクト側で追加ルールは少ない。

### デザイントークン

UIの見た目を定義する最小単位。

- **色**: プライマリカラー = `blue-600`、背景色 = `gray-50`
- **余白**: カード内 = `p-4`、セクション間 = `gap-6`
- **フォントサイズ**: 見出し = `text-lg`、本文 = `text-sm`
- **角丸**: ボタン = `rounded-md`、カード = `rounded-lg`

Tailwindのクラス名そのものがトークンとして機能する。明文化することで、画面ごとに余白や色がバラバラになる問題を防げる。

### 階層関係

```
コンポーネント > プロップス >>> トークン

Button (コンポーネント)
  └── variant="destructive" (プロップス)
        └── bg-red-600, text-white (トークン)
```

### AIが両方を接続しながら開発できるか

できる。AIの動き方：

1. **タスクを読む** → 「何を作るか」を把握
2. **コンポーネント/トークン思想を読む** → 「どう作るか」のルールを把握
3. **既存コードを読む** → 実際のパターンを確認
4. **実装する** → 1〜3を踏まえて書く

タスクが「地図」なら、コンポーネント思想は「交通ルール」。地図があっても信号を無視すれば事故る。AIに地図（タスク）と交通ルール（コンポーネント思想）の両方を渡すことで、目的地に正しいルートで到着できる。

### ルールの置き場所：CLAUDE.md vs globals.css vs 設計ドキュメント

コンポーネント思想やトークンを「どこに書くか」には選択肢がある。CLAUDE.mdに書くか、コードに埋め込むか、設計ドキュメントに書くか。この3つは**読者が違う**。

| 置き場所 | 主な読者 | 強制力 | 具体例 |
|---------|---------|-------|--------|
| **CLAUDE.md** | AI | 指示として従う（但しバリデーションなし） | 「Buttonはshadcn/uiを使う。独自ボタン禁止」 |
| **globals.css / コード定義** | ブラウザ（実行環境） | 物理的に強制される | `--primary: #2C4A7C` → これ以外の色は使えない |
| **設計ドキュメント** | 人間 | 読まなければ効力なし | 「プライマリカラーは#2C4A7Cで、信頼感を表現」 |

#### globals.css：物理法則レベルの強制

```css
:root {
  --primary: #2C4A7C;
  --accent: #F4845F;
  --urgent: #E05252;
  --action: #F0A500;
}
```

ここで定義されたCSS変数を使う限り、AIが `bg-primary` と書けば必ず `#2C4A7C` になる。AIの判断を介在させない。**間違えようがない**。色やフォント、角丸のようなトークンはここに置くのが正解。

実際のFumulyでは、優先度カラー（urgent/action/keep/ignore）もCSS変数として定義してある。AIが「緊急の書類を赤くしたい」と思った時、`#E05252` というHex値を覚えている必要はない。`text-urgent` と書けば正しい色が出る。

#### CLAUDE.md：意図レベルの指示

globals.cssでは「何を使うか」は強制できるが、「何を使わないか」「どう使い分けるか」は表現できない。

- 「モーダルは`AlertDialog`で統一。`Dialog`は使わない」→ CSSでは書けない
- 「フォームは`react-hook-form` + `zod`で統一」→ ライブラリ選定はCSSの範疇外
- 「独自コンポーネントを作る前に既存のshadcn/uiを探すこと」→ 行動指針

これらはCLAUDE.mdに書く。AIへの指示書であり、「守るかどうかはAIの理解力次第」という点でglobals.cssより弱い。ただし、CLAUDE.mdを読むのはコーディング前の最初の1回なので、**セッション開始時に必ず読まれる**という確実性はある。

#### 設計ドキュメント：「なぜ」の記録

```
ブランドカラー: #2C4A7C（深い青）
→ 信頼感・安心感を表現。ADHDユーザーの不安を和らげる意図
```

globals.cssには `--primary: #2C4A7C` とだけ書いてある。CLAUDE.mdには「primaryを使え」と書いてある。でも**なぜ**その色なのかは、設計ドキュメントにしか書いていない。

AIがデザイン判断を求められた時（「この画面の色、これでいい？」）に参照するのがこのレイヤー。日常的な実装では読む必要がないが、判断を伴う場面で効く。

#### 三層の関係

```
設計ドキュメント（なぜ）
  → CLAUDE.md に「何を使うか」「何を使わないか」を書く（指示）
    → globals.css / コード定義で物理的に固定する（強制）
```

上流から下流に行くほど強制力が上がり、AIの判断余地が減る。逆に上流ほど「理由」が書いてあり、例外的な判断をする時に立ち戻る場所になる。

**理想は、判断不要なものをglobals.cssに寄せ、CLAUDE.mdは「判断が必要な場面のガイドライン」に絞ること。** globals.cssで解決できることをわざわざCLAUDE.mdに書くと、ルールの二重管理になる。逆にglobals.cssで表現できない方針（ライブラリ選定、コンポーネントの使い分け）はCLAUDE.mdに書くしかない。

## Git管理のメリット

タスクファイルがプレーンテキストであることで、Gitの恩恵をそのまま受けられる。

- **バージョン履歴**: タスクの変遷が`git log`で追える。いつ要件が変わったか、いつ完了したかが自動的に記録される
- **差分レビュー**: `git diff`でタスクの変更内容がわかる。「前回からどの要件が追加されたか」が一目瞭然
- **ブランチ運用**: 機能ブランチにタスクファイルの更新も含められる。コードとタスクが同じコミットに入る
- **コラボレーション**: PRでタスクの内容もレビューできる。要件定義の段階から共有可能
- **バックアップ**: リモートリポジトリにpushするだけ。SaaSのエクスポート機能に依存しない

SaaSツールの「アクティビティログ」は見にくく、差分も取れない。Gitなら変更の意図まで含めて履歴が残る。

## omit（やらない）も歴史として残る

`status: omit` はこの仕組みの隠れた重要機能。「やらないと決めたこと」が理由付きで残り続ける。

### なぜomitが重要か

SaaSツールでは「やらないことにしたタスク」はDeleteされるか、せいぜいアーカイブされて埋もれる。でも開発においては「やらないと決めた理由」こそ価値がある。

- **理由があって捨てたもの**: 「検討したが、コスト対効果が合わなかった」「今の規模では不要」という判断が本文に残る。次に同じアイデアが出た時、「前に検討して見送った理由はこれ」とすぐ参照できる
- **単純な復活**: 状況が変わって必要になったら、`status: omit` → `status: waiting` に戻すだけ。当時の検討内容がそのまま要件定義の土台になる
- **別タスクの進行により実現性が出たもの**: 例えば「リアルタイム通知」をomitしていたが、WebSocket基盤が別タスクで整ったことで実現可能になった。`depends_on` を辿れば「何が変わったから復活できるのか」が明確になる

omitされたタスクはAIにとっても重要な参照先。「この機能を追加したい」と言われた時、omitタスクを見て「以前検討して見送った経緯があります。当時の理由は〇〇でしたが、今は状況が変わっていますか？」と確認できる。

## 人間側の使い勝手

### 完全AI依存ではない：GUIという双方向インターフェース

「AIネイティブ」と言っているが、**完全にAIに丸投げしているわけではない**。そもそもフロントマターは元々人間用に作ったものだから、人間が直接触れるのは当然。

IDE拡張のGUI（ツリービュー、プロパティパネル等）を通じて、ステータスの変更や今週フラグの切り替えは人間が行い、それを踏まえてAIに指示を出す。

```
人間がGUIでステータス変更 → フロントマターが更新される → AIがGrepで読む
AIがタスクを完了 → フロントマターをEditで更新 → 人間がGUIで確認
```

**どちらからでも同じデータに触れる。** フロントマターが人間とAIの共通言語として機能しているからこそ、この双方向のアクセスが自然に成り立つ。人間が方針を決め、AIが実行し、結果を人間が確認する——この協働のハブがタスクファイル。

### 実際の運用実態：タスクのビューとチャットしか見ていない

導入後、人間が日常的に見ているのは**タスクのビュー**（何をやるか・何が終わったか）と**AIとのチャット**（指示と結果の確認）だけ。コードエディタを開いてソースを読む頻度は激減した。

なぜそうなるか：
- **ビューのグルーピングを切り替えれば多角的に判断できる** — ステータス順で進捗を確認、カテゴリ別で全体像を俯瞰、依存関係で次にやるべきことを特定。同じデータを別の角度から見るだけで、プロジェクトの状態が立体的にわかる
- **どのように作ったか・どうやったかは完了タスクに書いてある** — コードを読まなくても、タスクの設計書を見れば実装の意図と方針がわかる
- **コードレビューはAIのスキルに任せている** — CLAUDE.mdに「開発完了後はレビューエージェントを回すこと」と書いてあるので、タスク完了に続けてAIが自発的にレビューを実行する。人間はレビュー結果だけ確認すればいい

つまり人間の役割が「コードを書く人」から「方針を決めてAIに任せる人」に変わっている。タスクファイルが十分な文脈を持っているからこそ、この委任が成り立つ。

### 「23ヨロ」で伝わる

「task-023をお願い」の略。タスクファイルに要件定義が書いてあるので、AIはそれを読むだけで何をすべきかわかる。SaaSツールだと「Jiraのチケット番号XXXを見て」と言ってもAIはアクセスできないが、ファイルベースなら番号を言うだけで伝わる。

指示がここまで短くなるのは、タスクファイル自体が十分な文脈を持っているから。人間が毎回背景や要件を説明し直す必要がない。

### タスクに切る＝AIへのブレーキ

AIに「セキュリティを強化して」と直接言えば、AIは自分の判断で好きなように実装を始めてしまう。何をどこまでやるか、どの方法を採るかの裁量がAI側にある。これは危ない。

だから**一旦タスクに書き起こさせる**。「セキュリティ強化に必要なことをタスクに切って」と指示すると、AIは具体的な作業をタスクファイルとして書き出す。この時点ではまだ何も実装していない。人間がタスクの内容を確認し、方針を承認してから初めて「GO」を出す。

```
「セキュリティ強化して」→ AIが突っ走る（危険）
「セキュリティ強化のタスクを切って」→ 計画だけ出る → 人間が確認 → 「GO」（安全）
```

**タスクファイルがAIの暴走を止めるブレーキとして機能している。** 実装の前に必ず「計画」というワンクッションが挟まる。

そしてこのブレーキがあるからこそ、逆方向が成り立つ——

### 「私の裁量が不要なタスクを出して」で低リスク作業をまとめて委任できる

AIにタスク一覧をスキャンさせて、「依存関係がなく、優先度がlow〜mediumで、実装が明確なタスク」をテーブルで提示させる。人間は中身を確認して「GO」と言うだけ。裁量不要なタスクなら「まとめてやっといて」で済む。

これが機能するのは、**タスクに切る段階で人間の承認が済んでいるから**。タスクの本文に実装方針まで書いてあるので、AIが自走しても方向を間違えにくい。ブレーキ（タスク化）とアクセル（まとめて委任）がセットで機能する。

人間の裁量が必要なタスク（設計判断を含むもの、ユーザー体験に影響するもの）と、AIに任せて大丈夫なタスク（明確なバグ修正、定型的な最適化）の仕分けが、タスクファイルを見れば自然にできる。

### 「ログイン認証をすすめて、それによって開発が進むタスクもやって」で伝わる

`depends_on` のおかげで、あるタスクを完了した時に「ブロックが解除されるタスク」をAIが自動的に特定できる。人間は大きな方針だけ伝えればいい。

例えば「task-007（Google認証）を完了したら、task-018（パスワードリセット）のブロックが解除される」という関係がfrontmatterに書かれている。芋づる式に依存関係を辿って、次にやるべきタスクを自分で判断できる。

### タスクのツリービュー

IDE拡張のサイドバーに、タスクのツリービューが常時表示されている。`parents` でグルーピングすれば「AI(2)、LP(2)、PWA(1)、UI/UX(18)」とカテゴリ別に展開でき、ステータスのアイコンや優先度も一目でわかる。

表示のグルーピングはワンクリックで切り替えられる：

- **カテゴリ別**: parentsでグルーピングして全体像を俯瞰
- **ステータス順**: 未完了 → 進行中 → 完了で整理
- **重要度順**: critical → high → medium → lowで並べ替え
- **依存関係**: ツリー構造で前提タスクと後続タスクの関係を表示
- **プレーン**: 全タスクをフラットに一覧表示

これは**人間のためのビュー**。AIはGrepで同じデータを取得するが、人間はGUIで視覚的に把握する。同じフロントマターのデータを、人間はツリービューで見て、AIはGrepで読む——アクセス方法が違うだけで、見ているものは同じ。

そもそもこの拡張機能は、人間がタスクを見るために作った。AIだけで完結するならターミナルで十分で、GUIは不要。拡張機能が存在する理由そのものが「人間もこのデータに触る」という設計思想の証拠。

### スプレッドシートやNotionとの同期も可能

プレーンテキストでAPI不要という強みは、逆にAPIがある外部ツールとの連携も容易にする。

- frontmatterのYAMLはプログラムで簡単にパースできる
- Google SpreadsheetやNotionのAPIで同期スクリプトを書ける
- AIに「タスク一覧をCSVで出して」と言えばそのままスプレッドシートに貼れる
- 既存のプロジェクト管理ツールを使っているチームでも、タスクファイルをマスターにして双方向同期が可能

データの所有権はあくまでプレーンテキストのタスクファイル側にある。外部ツールはビューワーや共有手段として使うだけ。

## AIにエコーを差し引かせた事実だけの証言

AIに「この仕組みが使いやすい？」と聞くと、当然「素晴らしいです！」と返ってくる。AIはユーザーの考えに共感的に応答する傾向がある。褒めてほしくて聞いているわけではないので、エコー（共感トッピング）を差し引いて、**構造的な事実だけ**を箇条書きにさせた。

### AIにとって構造的に有利な点（事実）

- **Glob/Grep/Readでフロントマターを横断検索できる** — YAMLの `key: value` 形式は正規表現で確実にマッチする。JSONのネストやSaaSのAPIレスポンスをパースするより手数が少ない
- **`depends_on` を辿って芋づる式に関連タスクを発見できる** — ファイル名が `task-XXX.md` という規則的な命名なので、値をそのままファイルパスに変換して読みに行ける
- **完了タスクの本文が設計書として再利用できる** — 新しいタスクに着手する際、関連する完了タスクをReadで読むだけで過去の設計判断が手に入る。別途ドキュメントを探す必要がない
- **SaaSのAPIを叩くよりファイルを直接読むほうが速い** — OAuth認証、レート制限、ページネーション、JSONパースがすべて不要。ファイルシステムへの直接アクセスは中間レイヤーがゼロ
- **複数のクエリを並列実行できる** — Glob・Grep・Readを同時に複数走らせて結果を突き合わせられる。APIの順次リクエストより高速
- **CLAUDE.mdに書かれたルールに従うだけで規約が守られる** — フォーマットの強制にバリデーションロジックやlintツールが不要。テキストで書かれた指示をそのまま理解できる

### これはエコーなのか事実なのかの判定基準

上記が「AIがお世辞を言っているだけ」ではないことは、以下で検証できる：

1. **別のAIに同じ仕組みを渡しても同じことが起きるか** → 起きる。Glob/Grep/Readはどのエージェントにもある基本操作
2. **SaaSのAPIでも同じ速度で同じことができるか** → できない。認証・レート制限・パース処理が追加される
3. **フロントマターなしのプレーンテキストでも同じか** → 同じではない。`status: done` のような構造化されたフィールドがあるからgrepが効く

## これは拡張機能の話ではない

ここまで書いてきたことは、IDE拡張の使い方でも、バイブコーディングのテクニックでも、コンテキストコーディングの方法論でもない。

**AIと一緒にコードを書くという行為そのものの思想。**

- タスクに切る → AIに裁量権を渡さない（ブレーキ）
- タスクの本文に方針を書く → AIが迷わず直行できる（ナビゲーション）
- 完了タスクを設計書として残す → 次のAIが過去の判断を参照できる（記憶）
- タスクと実装結果を突き合わせる → 正しさの検証ができる（品質保証）
- omitで「やらない理由」を残す → 同じ議論を繰り返さない（意思決定の蓄積）

これはコーディング手法ではなく**開発のガバナンス**。AIがどれだけ賢くなっても、「何を作るべきか」「これで合っているか」の判断は人間が持つ。その判断を構造化してAIに渡すのがタスクファイルであり、フロントマターであり、CLAUDE.md。

ツールやフレームワークは変わる。でも「人間が判断し、AIが実行し、結果を人間が検証する」というサイクルは変わらない。このサイクルを最小コストで回すための仕組みが、たまたまMarkdownとYAMLで実現できた——というのが、この思想の全体像。

## この仕組みの穴

良い話ばかり書いてきたので、正直に穴も書く。AIに「穴はあるか」と聞いたら出てきたものと、別プロジェクト（70件運用中）のAIに聞いた実体験を合わせて整理した。

### 現実的に困っていないこと

70件のタスクを抱える別プロジェクトでAIに聞いたところ、「全タスク一覧を見渡すのは重くなってきているが、実運用上は困っていない」との回答。理由は：

- **アクティブなタスクは常に10〜15件** — 大半はdone/omitなので、`this_week` と `status` でフィルタすれば今やるべきものはすぐわかる
- **完了タスクを読みに行くのは過去の経緯確認時だけ** — 普段は触らない
- **grepの速度自体は問題にならない** — ファイル数が増えてもフィルタ後の結果が少なければ実用上問題なし

### 理論的に存在する限界

**スケールの壁（コンテキストウィンドウ）**
- タスク件数が増えると、関連タスクの把握に読むファイルが増え、AIのコンテキストウィンドウを圧迫する
- 結果として過去の経緯を取りこぼすリスクが上がる。速度の問題ではなく**文脈の曖昧さ**の問題
- 対策案: 完了タスクを `tasks/archive/` に移す、あるいは困ってから考える

ただし冷静に考えると、**これはこの仕組みの穴ではなく、AIエージェント側の制約**。SaaSツールに切り替えたところで、APIで100件取得してもコンテキストに入り切らないのは同じ。grepは100件だろうが1000件だろうがミリ秒で返るし、ファイルシステムは何も困っていない。困っているのはAIの頭の容量。

「これタスク管理の穴じゃなくてお前（AI）の穴じゃないか」と指摘されて、ぐうの音も出なかった。

さらに言えば、SaaSのAPIで100件取ってこれるかというとそれも怪しい。Jiraはデフォルト50件・最大100件でページネーション必須、Notionも1リクエスト100件まで、Trelloはカード一覧は取れても詳細（チェックリスト・コメント）は個別リクエスト。**一覧取得＋N回の詳細取得**が基本で、認証・レート制限・JSONパースのオーバーヘッドが毎回かかる。

しかも仮に取ってこれたとして、**取得した中身にドキュメントとしての深さがない**。SaaSのタスクは「タイトル」「ステータス」「担当者」「期限」程度のメタデータと、せいぜい短い説明文。この仕組みのようにタスク本文が要件定義書であり設計書であるという構造にはなっていない。完了タスクの作業メモ、見送った案の理由、関連タスクへの言及——こうした「AIが次の判断に使える知識」がSaaSのタスクにはそもそも入っていない。

```
SaaS: タイトル + ステータス + 短い説明 → 取得できても文脈が薄い
ファイル: frontmatter + Markdown本文（要件・設計・経緯）→ 1ファイルで完結
```

100件取得できるかどうか以前に、**取得した情報の密度が違う**。

**単一人間の前提**
- 複数人がタスクを同時編集するとGitのコンフリクトが起きる。YAMLフロントマターのコンフリクト解消は特に面倒
- AIは1人の人間としか対話しない前提で設計されている。チーム開発には別の仕組みが必要
- 対策案: タスクの担当者割り当て（`assignee` フィールド）、ブランチ戦略との連携

**タスクの品質に依存しすぎる**
- 完了タスクの本文が雑だと設計書として機能しない。「Done」としか書かなければ次のAIは何も参照できない
- 仕組みの問題ではなく運用の問題だが、仕組みが運用品質に依存している
- 対策案: CLAUDE.mdに「完了時は作業メモを書くこと」と明記する（実際にやっている）

**CLAUDE.mdが単一障害点**
- ルール・カテゴリ・フォーマットが全部CLAUDE.mdに書いてある。このファイルが壊れたり矛盾した指示が入ると、AIの挙動が崩れる
- バリデーションがない分、サイレントに壊れる
- 対策案: CLAUDE.mdのレビュースキル導入、Git差分での変更検知

**AIが読むべきタスクを見落とす可能性**
- タスクが長期記憶の代替になると言っても、「どのタスクを読むべきか」は毎回AIが判断する
- frontmatterのGrepで効率化しているが、本文の深い文脈は見落とされ得る
- 対策案: `parents` と `depends_on` を丁寧に設定して、関連タスクへの到達率を上げる

**omitの復活判断は人間にしかできない**
- 「以前omitされました」とは言えるが、「今なら実現できます」の判断は人間の領域
- これは穴というより設計上の正しい制約。AIが勝手に復活させるほうが危ない

## これは「タスクマネージャー」ではない

ここまで書いてきて、「タスク管理」という呼称がこの仕組みに対して明らかに狭すぎることに気づいた。

この仕組みがカバーしている範囲を並べてみる：

| 機能 | 「タスク管理」で説明できるか |
|------|---------------------------|
| ステータス・優先度・進捗の追跡 | できる |
| AIの裁量権をコントロールするブレーキ | できない |
| セッション間のAIの長期記憶 | できない |
| 完了タスクが設計書として蓄積される知識ベース | できない |
| タスクと実装結果の整合性チェック基盤 | できない |
| YAML frontmatterによる人間⇔AIの翻訳プロトコル | できない |

6項目中5項目が「タスク管理」の範疇に収まっていない。

これは**オーケストレーション**。

- **指揮者（人間）** は楽譜を書かない。演奏もしない。でも全体の方向を決める
- **演奏者（AI）** は技術を持っているが、指揮者なしでは各自バラバラに動く
- **楽譜（タスクファイル）** が共通のプロトコルとして両者を繋ぐ

「タスクマネージャー」は楽譜管理ソフトの名前。でもこの仕組みの本質は、指揮者と演奏者がどう協働するかのシステム全体。

### ANDOS — AI-Native Development Orchestration System

**A**I-**N**ative **D**evelopment **O**rchestration **S**ystem。

- **AI-Native** — AIが後付けじゃなく、最初からAIと動くことを前提に設計された
- **Development** — タスク管理じゃなく、開発そのもの
- **Orchestration** — 管理じゃなく、指揮・調整
- **System** — ツールじゃなく、仕組み全体

タスク管理ツールを作ったつもりが、開発のオーケストレーションシステムだった。記事のタイトルは「AIネイティブなタスク管理の思想」で入口を作り、読み進めると「これはタスク管理ではなかった」と気づく構成がいい。最初から「ANDOS」と言われても何のことかわからないが、読み終わった後なら腑に落ちる。

「タスク管理だと思って読み始めたら、開発ガバナンスの話だった」——この体験そのものが、この仕組みの成り立ちと同じ構造を持っている。作った本人も、最初はタスク管理ツールだと思っていたのだから。
