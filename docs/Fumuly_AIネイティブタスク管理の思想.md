# AIネイティブタスク管理の思想

Fumulyで実際に41タスクを運用してきた中で見えてきた、AI駆動開発におけるタスク管理の考え方をまとめる。

---

## 基本構造

1ファイル = 1タスク（`tasks/task-XXX.md`）。YAML frontmatter + Markdown本文という、誰でも知ってるフォーマットの組み合わせ。

### frontmatterが提供する機能

| フィールド | 機能 | 用途 |
|-----------|------|------|
| `id` | 一意識別子 | タスク間の参照に使用 |
| `title` | タイトル | 一覧表示・検索 |
| `parents` | 複数カテゴリ | `[UI/UX, 機能]` のように複数の関心事を表現 |
| `status` | 6段階ステータス | idea → waiting → in-progress → done/omit/blocked |
| `priority` | 4段階優先度 | critical / high / medium / low |
| `depends_on` | 依存関係 | `[task-002, task-003]` で前提タスクを定義 |
| `this_week` | 週次スコープ | 今週やるかどうかのフラグ |
| `completed_at` | 完了日 | 時系列での振り返り |
| `progress` | 進捗率 | 0-100の数値 |
| `note` | 一行メモ | frontmatterだけ見て概要を把握 |
| `estimated_hours` | 見積もり | 工数管理 |

### Markdown本文の役割

- **未完了タスク** → 要件定義書（背景・ユースケース・実装案・検討した選択肢）
- **進行中タスク** → 作業コンテキスト（今何をしているか）
- **完了タスク** → 設計書（作業メモ・実装内容・見送った案とその理由）

タスクの「完了」は情報の消滅ではなく、**設計ドキュメントへの昇格**。

### タスクのライフサイクル

```
idea → waiting → in-progress → done
 (要件の種)   (要件定義書)    (作業中)     (設計書)
                ↘           ↕
               omit       blocked
           (やらない理由書)  (外因による待機)
```

どのフェーズでも本文に情報が蓄積され、完了後もプロジェクトの知識ベースとして機能し続ける。omitされたタスクも「なぜやらないか」の記録として残り、状況が変われば復活できる。blockedは外因（Stripeの審査待ち、他サービスの承認待ち、ボスの確認待ち等）による一時停止で、解除されればin-progressに戻る。

## 既存ツールにない特徴

### 複数親カテゴリ

`parents: [セキュリティ, UI/UX]` で1タスクが複数の関心事に属する。多くのツールは1カテゴリか、タグ機能があっても検索が弱い。これがあるから「セキュリティ関連のタスク」でも「UI/UX関連のタスク」でも、どちらからでもそのタスクにたどり着ける。

### グルーピング能力

| 軸 | 方法 |
|----|------|
| カテゴリ別 | `parents` でgrep |
| ステータス別 | `status` でgrep |
| 優先度別 | `priority` でgrep |
| 依存関係別 | `depends_on` で芋づる式に辿る |
| 週次スコープ | `this_week` でgrep |
| 時系列 | `completed_at` でソート |

これらを自由に組み合わせてクロス集計できる。「セキュリティ関連で未完了のhighタスクは？」のような質問にAIが数秒で答えられる。

### AIネイティブ

人間もAIも同じ方法（ファイルの読み書き）でアクセスする。SaaSツールのようにGUIとAPIで別々の入口を用意する必要がない。

### CLAUDE.mdによる規約定義

フォーマット・カテゴリ一覧・運用ルールをCLAUDE.mdに書くだけで、AIが従う。設定画面もプラグインも不要。

### ゼロ依存

プレーンテキストなのでベンダーロックインがない。Git管理でバージョン履歴も自動的に残る。

## AIがタスクをどう読み、どう探すか（技術的な裏側）

AIコーディングエージェント（Claude Code等）は、内部的に以下の3種類のツールを使ってファイルシステムを操作する。SaaSのAPIではなく、開発者がターミナルで行う操作と本質的に同じこと。

### AIが使う3つの基本ツール

| ツール | 対応するコマンド | 用途 |
|--------|-----------------|------|
| **Glob** | `find`, `ls` | ファイル名のパターン検索。`tasks/task-*.md` で全タスクファイルを一覧取得 |
| **Grep** | `grep`, `rg` | ファイル内容の正規表現検索。frontmatterの特定フィールドを横断検索 |
| **Read** | `cat` | ファイル内容の読み込み。タスク本文の詳細を取得 |

重要なのは、**この3つだけで全操作が完結する**こと。専用のクエリ言語もAPI認証も不要。

### 実際の動き：全体把握

AIがまず最初にやることは、Globでタスクファイルの一覧を取得すること。

```
Glob("tasks/task-*.md")
→ task-001.md, task-002.md, ... task-042.md （42件）
```

次にGrepでfrontmatterだけを一括スキャンする。

```
Grep("^status:", path="tasks/")
→ task-001.md: status: done
→ task-005.md: status: waiting
→ task-006.md: status: omit
→ task-014.md: status: waiting
→ ...（42件分のステータスが一瞬で返る）
```

42ファイルあっても、Markdown本文は読まない。frontmatterだけなら数秒で全タスクの状態を把握できる。**必要なものだけ深掘りする**のがポイント。

### 実際の動き：絞り込み

「セキュリティ系で未完了のタスクある？」と聞かれたら、AIは内部的にこう動く。

```
Grep("^parents:.*セキュリティ", path="tasks/")
→ task-004.md: parents: [セキュリティ]
→ task-007.md: parents: [セキュリティ]
→ task-015.md: parents: [セキュリティ]
→ task-018.md: parents: [セキュリティ, UI/UX]
→ task-034.md: parents: [セキュリティ]
→ task-042.md: parents: [セキュリティ, バグ修正]
```

ここで6件ヒット。次にこれらのステータスを確認して、`status: done`以外のものを抽出する。人間がダッシュボードでフィルタする操作を、AIはGrepで行う。結果は同じだが、APIキーもOAuth認証も不要。

**そしてここが重要：これらのツール呼び出しは並列実行できる。** Glob、Grep（セキュリティ検索）、Grep（ステータス検索）を同時に走らせて結果を突き合わせる。SaaSのAPIだとリクエストを順番に送る必要があるが、ファイルシステムは並列アクセスに制限がない。

### 実際の動き：深掘り

絞り込みの結果、気になるタスクが見つかったらReadで本文を読む。

```
Read("tasks/task-034.md")
→ frontmatter + Markdown本文がすべて返る
→ 「task-027と合わせてプライバシーリスクを軽減」という記述を発見
```

ここで本文中の `task-027` という文字列を見つけたら、次はそのファイルを読みに行く。

```
Read("tasks/task-027.md")
→ 会話履歴の保持期間ポリシーの設計書が出てくる
→ さらに task-021（プライバシーポリシー）への言及を発見
```

これが**芋づる式探索**。`depends_on` と本文中の `task-XXX` 参照をリンクとして辿る。人間にはただの文字列だが、AIにとっては**実質的なハイパーリンク**。

### ツール呼び出しの実行環境

これらのツールは、AIコーディングエージェントの実行環境によって提供される。

```
ユーザーの発話
  ↓
AIエージェント（Claude等）
  ↓ ツール呼び出し
┌─────────────────────────┐
│ エージェント実行環境      │
│  Glob → ファイルシステム  │
│  Grep → ファイルシステム  │
│  Read → ファイルシステム  │
│  Edit → ファイルシステム  │
│  Bash → シェル            │
│  Write → ファイルシステム │
└─────────────────────────┘
  ↓ 結果
AIが結果を解釈して回答
```

SaaSツールだと、このフローが以下に変わる：

```
ユーザーの発話
  ↓
AIエージェント
  ↓ MCP接続 or API呼び出し
┌─────────────────────────┐
│ 外部サービス              │
│  OAuth認証                │
│  API rate limit           │
│  JSONレスポンスのパース    │
│  ページネーション処理     │
└─────────────────────────┘
  ↓ 結果
AIが結果を解釈して回答
```

間に挟まるレイヤーが増えるほど、遅くなるし壊れやすくなる。ファイルベースだとこの中間レイヤーが**ゼロ**。

### 完了タスクを設計書として参照

新しいタスクに取り組む時、AIは関連する完了タスクをReadで読む。例えばtask-041（アーカイブ機能）に着手する際：

```
Read("tasks/task-027.md") → 会話履歴の件数ベース削除の実装パターンを参考に
Read("tasks/task-024.md") → select最適化の方針との整合性を確認
Read("tasks/task-021.md") → プライバシーポリシーとの法的整合性を確認
```

**完了タスクの作業メモが、次のタスクの設計判断に直接使われる。** SaaSツールの完了タスクはアーカイブされて二度と見られないが、この仕組みでは蓄積された知識として活用し続ける。

### セッションを跨いだ記憶の代替

AIは毎回セッションがリセットされる。でもタスクファイルをGrepとReadで読めば：

- **何が終わっていて**（`Grep("^status: done")` → 39件）
- **なぜその設計にしたか**（`Read` で本文の検討経緯を取得）
- **何がまだ残っていて**（`Grep("^status: waiting")` → 2件）
- **何に依存しているか**（`Grep("^depends_on:")` で依存関係マップ）

がすべてわかる。**タスクファイルがAIの長期記憶として機能している。**

## AIが関連タスクの更新漏れに気づける仕組み

例えばtask-041（アーカイブ機能と30日自動クリーンアップ）を実装した時：

1. task-041の本文に「30日でデータ削除」という仕様がある
2. `parents: [UI/UX, 法的文書]` → 法的文書カテゴリから関連タスクを探す
3. task-021（プライバシーポリシー修正）を見つける → 完了済みだが、**当時の修正内容にはまだ30日ルールが反映されていない**
4. task-027（会話履歴の保持期間）も見つける → こちらも関連

この時点で「プライバシーポリシーに30日自動削除の記述を追加する必要があります」と提案できる。

これができるのは、完了タスクの本文に**何を修正したか**が残っているから。task-021を読めば「第6条にアカウント削除の記載を追加した」ことがわかるので、「でも30日自動削除のことはまだ書かれていない」という差分に気づける。

## AIがタスクを追加する時の思考プロセス

「セキュリティの向上をタスクに切って」と言われた時の動き方：

### Step 1: 既存のセキュリティタスクを全部読む

`parents` に `セキュリティ` を含むタスクを探す。

- task-004（セキュリティ）→ 完了済み。何をやったか本文で確認
- task-007（セキュリティ）→ Google認証追加。完了済み
- task-015（セキュリティ）→ 完了済み。内容確認
- task-018（セキュリティ, UI/UX）→ パスワードリセット。完了済み
- task-034（セキュリティ）→ 暗号化。完了済み

**完了タスクの本文を読んで、「何が既に対策済みか」を把握する。** これをやらないと、既にやったことを重複して提案してしまう。

### Step 2: 関連カテゴリも横断チェック

セキュリティに関係しそうな他カテゴリも見る。

- task-021（法的文書）→ プライバシーポリシー修正。認証データの削除に言及
- task-027（法的文書, バックエンド）→ 会話履歴の自動削除。暗号化と併用
- task-041（UI/UX, 法的文書）→ 30日自動クリーンアップ

**`parents` の複数カテゴリ機能があるから、カテゴリを横断して関連タスクを発見できる。**

### Step 3: コードベースの現状を確認

タスクだけでなく、実際のコードも見る。API認証、RLS、入力バリデーション、環境変数の扱いなど。

### Step 4: 「まだやっていないこと」を特定

既存タスクの完了内容（設計書）とコードの現状を突き合わせて、**差分＝やるべきこと**を導き出す。

### Step 5: タスクファイルを作成

```yaml
---
id: task-043
title: APIレート制限の強化
parents: [セキュリティ, バックエンド]
status: waiting
priority: medium
depends_on: [task-034]
this_week: false
completed_at: null
progress: 0
note: "解析APIのレート制限を強化。既存のIP制限に加えユーザー単位の制限を追加"
estimated_hours: 2
---
```

ポイント：
- `parents: [セキュリティ, バックエンド]` → 複数カテゴリで適切に分類
- `depends_on: [task-034]` → 暗号化タスクとの関連を明示。次にこのタスクを拾ったAIが「task-034で暗号化済みだからそこは考慮不要」と判断できる
- 本文にはユースケース・背景・実装案・検討した選択肢 → 未完了タスク＝要件定義書になる

## 重複コードの防止

### なぜ重複が起きるか

バイブコーディングもコンテキストコーディングも、AIは**今見えているファイルだけ**で判断する。「この処理、別のファイルにもう書いてあるよ」という情報がないから、同じようなバリデーション、同じようなエラーハンドリング、同じようなAPIコールを何度も書いてしまう。

### タスクファイルがどう効くか

「書類保存時にバリデーションを追加して」と指示された時：

1. task-019（DB操作のエラーハンドリング）を読む → 「各所で `{ error }` を確認し、失敗時にユーザーへ通知」というパターンが既にある
2. task-023（JSON.parseのtry/catch）を読む → エラーハンドリングの日本語メッセージ方針がわかる
3. task-025（重複スキャン検知）を読む → 保存前チェックのパターンが既にある

**既存のパターンに合わせる判断ができる。** 新しい関数を書くのではなく、既存のやり方を踏襲する。

### ゼロにはならない理由

- タスクの本文に書かれていない暗黙的なパターン（コードの書き方の癖など）は拾えない
- タスク化されていない小さな実装（ユーティリティ関数など）は見落とす
- AIのコンテキストウィンドウの限界で、全タスクを一度に読めない場合がある

### 限りなく抑えるための条件

完了タスクの本文に以下が書かれていれば、重複はかなり抑えられる：

1. **どのファイルに何を実装したか**（実装箇所）
2. **どういうパターンで書いたか**（設計判断）
3. **なぜその方法を選んだか**（見送った案）

**完了タスクの品質 = 重複防止の精度。** 雑に「Done」とだけ書いてあるタスクだと効果は薄いが、作業メモがしっかり書かれていれば、重複はかなり抑えられる。

## コンポーネント思想・トークン思想との接続

タスク管理が「なぜ作ったか・何を作ったか」の記録なら、コンポーネント思想は「どう作るか」のルールブック。この2つは補完関係にある。

### コンポーネント思想

UIの部品の選定・使い方ルール。

- 「Buttonは`shadcn/ui`の`Button`を使う。独自ボタンは作らない」
- 「モーダルは`AlertDialog`で統一。`Dialog`は使わない」
- 「フォームは`react-hook-form` + `zod`で統一」

これがあると、AIが新機能を実装する際に既存コンポーネントを再利用する。ないと、同じようなボタンやダイアログを毎回ゼロから書いてしまう。

### プロップス

コンポーネントの振る舞い・バリエーション。

- `variant="destructive"` → 削除系のアクション
- `size="sm"` → コンパクトな表示

shadcn/uiが定義済みなので、プロジェクト側で追加ルールは少ない。

### デザイントークン

UIの見た目を定義する最小単位。

- **色**: プライマリカラー = `blue-600`、背景色 = `gray-50`
- **余白**: カード内 = `p-4`、セクション間 = `gap-6`
- **フォントサイズ**: 見出し = `text-lg`、本文 = `text-sm`
- **角丸**: ボタン = `rounded-md`、カード = `rounded-lg`

Tailwindのクラス名そのものがトークンとして機能する。明文化することで、画面ごとに余白や色がバラバラになる問題を防げる。

### 階層関係

```
コンポーネント > プロップス >>> トークン

Button (コンポーネント)
  └── variant="destructive" (プロップス)
        └── bg-red-600, text-white (トークン)
```

### AIが両方を接続しながら開発できるか

できる。AIの動き方：

1. **タスクを読む** → 「何を作るか」を把握
2. **コンポーネント/トークン思想を読む** → 「どう作るか」のルールを把握
3. **既存コードを読む** → 実際のパターンを確認
4. **実装する** → 1〜3を踏まえて書く

タスクが「地図」なら、コンポーネント思想は「交通ルール」。地図があっても信号を無視すれば事故る。AIに地図（タスク）と交通ルール（コンポーネント思想）の両方を渡すことで、目的地に正しいルートで到着できる。

## Git管理のメリット

タスクファイルがプレーンテキストであることで、Gitの恩恵をそのまま受けられる。

- **バージョン履歴**: タスクの変遷が`git log`で追える。いつ要件が変わったか、いつ完了したかが自動的に記録される
- **差分レビュー**: `git diff`でタスクの変更内容がわかる。「前回からどの要件が追加されたか」が一目瞭然
- **ブランチ運用**: 機能ブランチにタスクファイルの更新も含められる。コードとタスクが同じコミットに入る
- **コラボレーション**: PRでタスクの内容もレビューできる。要件定義の段階から共有可能
- **バックアップ**: リモートリポジトリにpushするだけ。SaaSのエクスポート機能に依存しない

SaaSツールの「アクティビティログ」は見にくく、差分も取れない。Gitなら変更の意図まで含めて履歴が残る。

## omit（やらない）も歴史として残る

`status: omit` はこの仕組みの隠れた重要機能。「やらないと決めたこと」が理由付きで残り続ける。

### なぜomitが重要か

SaaSツールでは「やらないことにしたタスク」はDeleteされるか、せいぜいアーカイブされて埋もれる。でも開発においては「やらないと決めた理由」こそ価値がある。

- **理由があって捨てたもの**: 「検討したが、コスト対効果が合わなかった」「今の規模では不要」という判断が本文に残る。次に同じアイデアが出た時、「前に検討して見送った理由はこれ」とすぐ参照できる
- **単純な復活**: 状況が変わって必要になったら、`status: omit` → `status: waiting` に戻すだけ。当時の検討内容がそのまま要件定義の土台になる
- **別タスクの進行により実現性が出たもの**: 例えば「リアルタイム通知」をomitしていたが、WebSocket基盤が別タスクで整ったことで実現可能になった。`depends_on` を辿れば「何が変わったから復活できるのか」が明確になる

omitされたタスクはAIにとっても重要な参照先。「この機能を追加したい」と言われた時、omitタスクを見て「以前検討して見送った経緯があります。当時の理由は〇〇でしたが、今は状況が変わっていますか？」と確認できる。

## 人間側の使い勝手

### 実際の運用実態：タスクのビューとチャットしか見ていない

導入後、人間が日常的に見ているのは**タスクのビュー**（何をやるか・何が終わったか）と**AIとのチャット**（指示と結果の確認）だけ。コードエディタを開いてソースを読む頻度は激減した。

なぜそうなるか：
- **ビューのグルーピングを切り替えれば多角的に判断できる** — ステータス順で進捗を確認、カテゴリ別で全体像を俯瞰、依存関係で次にやるべきことを特定。同じデータを別の角度から見るだけで、プロジェクトの状態が立体的にわかる
- **どのように作ったか・どうやったかは完了タスクに書いてある** — コードを読まなくても、タスクの設計書を見れば実装の意図と方針がわかる
- **コードレビューはAIのスキルに任せている** — CLAUDE.mdに「開発完了後はレビューエージェントを回すこと」と書いてあるので、タスク完了に続けてAIが自発的にレビューを実行する。人間はレビュー結果だけ確認すればいい

つまり人間の役割が「コードを書く人」から「方針を決めてAIに任せる人」に変わっている。タスクファイルが十分な文脈を持っているからこそ、この委任が成り立つ。

### 「23ヨロ」で伝わる

「task-023をお願い」の略。タスクファイルに要件定義が書いてあるので、AIはそれを読むだけで何をすべきかわかる。SaaSツールだと「Jiraのチケット番号XXXを見て」と言ってもAIはアクセスできないが、ファイルベースなら番号を言うだけで伝わる。

指示がここまで短くなるのは、タスクファイル自体が十分な文脈を持っているから。人間が毎回背景や要件を説明し直す必要がない。

### 「私の裁量が不要なタスクを出して」で低リスク作業を委任できる

AIにタスク一覧をスキャンさせて、「依存関係がなく、優先度がlow〜mediumで、実装が明確なタスク」をテーブルで提示させる。人間は中身を確認して「GO」と言うだけ。

これが機能するのは、タスクの本文に実装方針まで書いてあるから。AIが自走しても方向を間違えにくい。人間の裁量が必要なタスク（設計判断を含むもの、ユーザー体験に影響するもの）と、AIに任せて大丈夫なタスク（明確なバグ修正、定型的な最適化）の仕分けが自然にできる。

### 「ログイン認証をすすめて、それによって開発が進むタスクもやって」で伝わる

`depends_on` のおかげで、あるタスクを完了した時に「ブロックが解除されるタスク」をAIが自動的に特定できる。人間は大きな方針だけ伝えればいい。

例えば「task-007（Google認証）を完了したら、task-018（パスワードリセット）のブロックが解除される」という関係がfrontmatterに書かれている。芋づる式に依存関係を辿って、次にやるべきタスクを自分で判断できる。

### タスクのツリービュー

AIにタスク一覧を出力させる時、表示方法を自由に選べる：

- **ステータス順**: 未完了 → 進行中 → 完了で整理
- **重要度順**: critical → high → medium → lowで並べ替え
- **依存関係**: ツリー構造で前提タスクと後続タスクの関係を表示
- **カテゴリ別**: parentsでグルーピングして全体像を俯瞰
- **プレーン**: 全タスクをフラットに一覧表示

SaaSツールではビューの切り替えに設定画面の操作が必要だが、AIに「依存関係で見せて」と言うだけで切り替わる。表示方法自体がプログラマブル。

### スプレッドシートやNotionとの同期も可能

プレーンテキストでAPI不要という強みは、逆にAPIがある外部ツールとの連携も容易にする。

- frontmatterのYAMLはプログラムで簡単にパースできる
- Google SpreadsheetやNotionのAPIで同期スクリプトを書ける
- AIに「タスク一覧をCSVで出して」と言えばそのままスプレッドシートに貼れる
- 既存のプロジェクト管理ツールを使っているチームでも、タスクファイルをマスターにして双方向同期が可能

データの所有権はあくまでプレーンテキストのタスクファイル側にある。外部ツールはビューワーや共有手段として使うだけ。

## SaaSツールとの比較まとめ

| 観点 | SaaSツール | この仕組み |
|------|-----------|-----------|
| データアクセス | API経由でJSON取得 | ファイルを直接読む |
| 認証 | OAuthトークン管理 | 不要 |
| クエリ | ツール固有のクエリ言語 | grep / ファイル読み込み |
| 完了タスク | アーカイブされて埋もれる | 設計書として蓄積 |
| ドキュメント | タスクとは別管理（Confluence等） | 同一ファイルに統合 |
| セッション跨ぎ | 別途仕組みが必要 | タスクファイル自体が記憶 |
| バージョン管理 | ツール内のアクティビティログ | Gitの完全な履歴管理 |
| ベンダーロックイン | あり | なし（プレーンテキスト） |
| AI連携 | MCP接続やAPI連携が必要 | 何も設定せずにそのまま読める |
